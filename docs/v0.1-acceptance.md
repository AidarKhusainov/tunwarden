# v0.1 proxy-only acceptance checklist

This document is the manual release-gate checklist for `v0.1.0: proxy-only technical preview`.

The checklist verifies that the implemented v0.1 commands form one coherent proxy-only flow on a real Linux host and that the flow does not mutate host networking state.

## Scope

In scope:

- local CLI checks;
- daemon startup through `go run ./cmd/tunwardend` or the documented systemd unit path;
- VLESS share URI import;
- Base64 URI-list subscription import from `file://`, `http://`, or `https://` sources;
- proxy-only Xray plan, connect, status, logs, disconnect, and recovery dry-run;
- verification that TUN, routes, DNS, nftables, and firewall state are not changed by proxy-only mode.

Out of scope for v0.1:

- TUN mode;
- route mutation;
- DNS mutation;
- nftables/firewall mutation;
- `recover --execute --yes`;
- reconnect state machine;
- automatic Xray download or update;
- packaging beyond the existing manual service/unit foundation.

## Required test host

Run this checklist on at least one Tier 1 Linux target before declaring `v0.1.0` complete:

- Ubuntu LTS desktop, or
- Debian stable desktop.

The host should have:

- Go 1.26.4, matching `go.mod` and CI;
- `systemd` and `journalctl`;
- a locally installed `xray` binary, or an explicit `TUNWARDEN_XRAY_PATH` pointing to an executable Xray binary;
- a supported VLESS share URI or a Base64 URI-list subscription containing at least one supported VLESS entry.

## Secret-backed acceptance inputs

For real acceptance runs, repository or environment secrets may provide the import targets without committing provider data to the repository.

Known secret names:

- `NLD_LINK`: supported VLESS share URI used for single-profile import checks.
- `TEST_USER_SUBSCRIBER_LINK`: HTTPS Base64 URI-list subscription URL used for subscription import checks.

Agents and workflows must treat these as secret inputs:

- never commit their values;
- never print them directly;
- never include them in PR comments, logs, test fixtures, screenshots, or verification records;
- pass them only through environment variables or a local secret manager;
- redact command transcripts before sharing them publicly.

Recommended local environment wiring:

```bash
export TUNWARDEN_ACCEPTANCE_VLESS_URI="$NLD_LINK"
export TUNWARDEN_ACCEPTANCE_SUBSCRIPTION_URL="$TEST_USER_SUBSCRIBER_LINK"
```

Recommended import checks:

```bash
go run ./cmd/tunwarden import "$TUNWARDEN_ACCEPTANCE_VLESS_URI"
go run ./cmd/tunwarden import "$TUNWARDEN_ACCEPTANCE_SUBSCRIPTION_URL"
```

When using GitHub Actions, expose these values only through `secrets.NLD_LINK` and `secrets.TEST_USER_SUBSCRIBER_LINK` in manually reviewed workflows. GitHub-hosted CI can validate real import/plan behavior with these secrets, but the complete release gate still requires a Tier 1 Linux host with Xray and strict before/after networking snapshots.

## Safety snapshot

Capture the host networking state before and after the proxy-only flow. The two snapshots must not show TunWarden-created TUN devices, route changes, DNS changes, nftables changes, or firewall changes.

The release verification record is incomplete if firewall state cannot be inspected. Do not hide missing permissions or missing tools with `|| true` for nftables/firewall snapshots.

Required read-only snapshot commands:

```bash
ip link show
ip route show table all
ip rule show
resolvectl status || true
sudo nft -a list ruleset
```

Also capture host firewall frontend state when present:

```bash
if command -v ufw >/dev/null 2>&1; then
  sudo ufw status verbose
  sudo ufw show raw
fi

if command -v iptables-save >/dev/null 2>&1; then
  sudo iptables-save
fi

if command -v ip6tables-save >/dev/null 2>&1; then
  sudo ip6tables-save
fi
```

`resolvectl status || true` is acceptable only because some Tier 1 installations may not use `systemd-resolved`; the verification record must still say whether DNS state was inspectable. Firewall snapshots must succeed, or the record must explicitly say that no release-gate conclusion about nftables/firewall mutation can be made on that host.

Save before/after output outside the repository or attach it to the release verification record. Redact provider domains, tokens, user identities, and subscription URLs before sharing logs publicly.

## Automated checks

From the repository root:

```bash
test -z "$(gofmt -l .)"
go test ./...
go run ./cmd/tunwarden version
go run ./cmd/tunwarden doctor
go run ./cmd/tunwarden recover
```

Expected result:

- formatting is clean;
- tests pass;
- `version` exits `0`;
- `doctor` completes with either healthy output or actionable read-only warnings;
- `recover` is read-only and does not remove files, kill processes, or mutate networking.

## Manual proxy-only flow

Use a temporary user state directory when possible so that the acceptance run is repeatable:

```bash
export XDG_STATE_HOME="$(mktemp -d)"
export TUNWARDEN_XRAY_PATH=/path/to/xray
```

Choose one or both real import targets from the secret-backed inputs:

```bash
export TUNWARDEN_ACCEPTANCE_IMPORT_TARGET="$TUNWARDEN_ACCEPTANCE_VLESS_URI"
# or:
export TUNWARDEN_ACCEPTANCE_IMPORT_TARGET="$TUNWARDEN_ACCEPTANCE_SUBSCRIPTION_URL"
```

Start the daemon in one terminal:

```bash
go run ./cmd/tunwardend
```

In another terminal, run:

```bash
go run ./cmd/tunwarden status
go run ./cmd/tunwarden import "$TUNWARDEN_ACCEPTANCE_IMPORT_TARGET"
go run ./cmd/tunwarden profile list
go run ./cmd/tunwarden subscription list
go run ./cmd/tunwarden plan --mode proxy-only <profile-id>
go run ./cmd/tunwarden doctor --core --xray "$TUNWARDEN_XRAY_PATH"
go run ./cmd/tunwarden connect --mode proxy-only <profile-id>
go run ./cmd/tunwarden status
go run ./cmd/tunwarden logs
go run ./cmd/tunwarden logs --core
go run ./cmd/tunwarden disconnect
go run ./cmd/tunwarden status
go run ./cmd/tunwarden recover
```

Expected result:

- `status` is inactive before connect;
- `import` stores either one VLESS profile or one Base64 subscription plus supported subscription-owned VLESS profiles;
- unsupported subscription entries are reported without leaking full share URIs or user identities;
- `profile list` shows the imported profile IDs needed by `plan` and `connect`;
- `subscription list` shows imported subscription metadata when the import target was a subscription;
- `plan --mode proxy-only` is read-only and explicitly says that TUN, routes, DNS, nftables, and firewall state are not modified;
- `doctor --core --xray <path>` validates the explicit Xray binary without starting a long-running connection;
- `connect --mode proxy-only <profile-id>` starts only daemon-managed Xray process/runtime config state;
- active `status` reports `Mode: proxy-only`, proxy listeners, `TUN: disabled`, `Routes: not modified`, `DNS: not modified`, and `Firewall: not modified`;
- `logs` and `logs --core` are read-only and redacted;
- `disconnect` stops the daemon-managed Xray process and removes normal generated runtime config;
- final `status` is inactive;
- final `recover` remains dry-run only and reports no required cleanup after a clean disconnect.

## Additional failure checks

Run these negative checks before release:

```bash
go run ./cmd/tunwarden recover --execute --yes
go run ./cmd/tunwarden status --json
go run ./cmd/tunwarden import --json "$TUNWARDEN_ACCEPTANCE_VLESS_URI"
go run ./cmd/tunwarden connect --mode tun <profile-id>
```

Expected result:

- `recover --execute --yes` fails fast because cleanup execution is deferred in v0.1;
- `status --json` fails fast with invalid usage until its JSON contract is implemented;
- `import --json` fails fast with invalid usage until its JSON contract is implemented;
- TUN mode is rejected and does not create interfaces, routes, DNS config, nftables state, or firewall state.

## Known v0.1 limitations

These limitations are acceptable for `v0.1.0: proxy-only technical preview` only when they are visible in release notes:

- proxy-only mode does not provide full-tunnel leak protection;
- TUN, route, DNS, nftables, and firewall behavior are intentionally absent;
- `recover` is dry-run only;
- `status --json`, `logs --json`, and top-level `import --json` are deferred;
- top-level `import` supports VLESS share URIs and Base64 URI-list subscriptions only;
- subscription import supports VLESS entries only; VMess, Trojan, and Shadowsocks import are deferred;
- subscription delete, scheduled updates, update dry-run, and provider metadata are deferred;
- Xray must be installed by the user or provided through `TUNWARDEN_XRAY_PATH`.

## Verification record

A release verification record should include:

- distribution name and version;
- kernel version;
- Go version;
- Xray version and path;
- commit SHA under test;
- automated check results;
- which secret-backed input type was used: VLESS URI, subscription URL, or both;
- manual command transcript with secrets redacted;
- before/after networking snapshots;
- explicit statement that TUN, route, DNS, nftables, UFW, iptables, and ip6tables snapshots were inspectable;
- confirmation that no TUN, route, DNS, nftables, or firewall mutation occurred;
- any remaining limitations or follow-up issues.

This repository change defines the checklist and keeps it runnable. The release owner must still attach a completed verification record before tagging `v0.1.0`.
