# v0.2 safe TUN preview acceptance checklist

This document defines the manual release-gate checklist for `v0.2.0: safe TUN preview`.

The checklist verifies that the accumulated v0.2 behavior works together on a real Tier 1 Linux host and follows the CLI, networking reliability, and state/security contracts. It is intentionally a release gate, not a unit-test substitute.

A documentation-only change that adds or edits this checklist must not close the release-gate issue. Close the acceptance issue only when a completed redacted verification record is attached to the release PR, release notes, this document, or the issue thread. Do not mark this checklist as passed from CI-only results or from a host where TUN, route, DNS, nftables, runtime, and process state could not be inspected.

## Scope

In scope:

- `go test ./...` for the exact commit under test;
- read-only `doctor`, `plan --mode tun`, `status`, and `recover` behavior;
- successful full-tunnel `connect --mode tun` and `disconnect` on one Tier 1 Linux desktop target;
- daemon-owned TUN interface, route, policy-rule, systemd-resolved DNS, nftables, Xray, TUN adapter, transaction, rollback, and cleanup behavior;
- rollback after at least one forced post-apply failure;
- VPN server route bypass outside `tunwarden0`;
- confirmation that `recover` without `--execute` is non-mutating;
- verification that the dedicated explicit recovery cleanup issue has already passed before this release gate closes.

Out of scope for v0.2 acceptance:

- packaging acceptance;
- suspend/resume, Wi-Fi roaming, reconnect loops, and NetworkManager event handling;
- non-systemd DNS fallback;
- GUI behavior;
- broad protocol expansion beyond profiles already supported by the current import path.

## Required test host

Run this checklist on at least one Tier 1 target:

- Ubuntu LTS desktop, or
- Debian stable desktop.

The host must have:

- Go toolchain selected from the `go` and `toolchain` lines in `go.mod`, verified with `go version`;
- `systemd`, `NetworkManager`, and `systemd-resolved` when validating DNS apply/rollback;
- `iproute2`, `nft`, and Linux `/dev/net/tun`;
- an executable Xray binary;
- an executable `tun2socks` binary, or `TUNWARDEN_TUN2SOCKS_PATH` pointing to one;
- a supported profile or import target that can establish an outbound connection.

The packaged baseline daemon remains intentionally unprivileged. For this manual TUN preview run, use a development-only privileged daemon deployment that keeps the daemon/core process model auditable:

- do not run `tunwardend` as root for the success path;
- run the daemon as a dedicated unprivileged service user;
- grant only the networking privileges needed for the TUN preview, currently `CAP_NET_ADMIN`-equivalent rights for TUN, route, policy-rule, DNS link, and nftables changes;
- expose `TUNWARDEN_XRAY_PATH` and `TUNWARDEN_TUN2SOCKS_PATH` to the daemon environment;
- record the exact systemd unit, drop-in, or development launcher used for the verification.

If the daemon is run as root, TUN-mode core startup must fail. That is a safety check, not an acceptable success-path deployment.

## Secret-backed acceptance inputs

Real provider inputs must stay out of the repository.

Recommended local environment variables:

```bash
export TUNWARDEN_ACCEPTANCE_IMPORT_TARGET="$NLD_LINK"
# or:
export TUNWARDEN_ACCEPTANCE_IMPORT_TARGET="$TEST_USER_SUBSCRIBER_LINK"
```

Rules:

- never commit share URIs, subscription URLs, UUIDs, provider tokens, passwords, private keys, or generated Xray configs;
- never paste raw command output containing provider data into GitHub;
- redact profile IDs only when they encode secrets; deterministic non-secret IDs may be recorded;
- record whether a direct share URI, subscription URL, or pre-existing local profile was used.

## Automated checks

From the repository root for the exact commit under test:

```bash
test -z "$(gofmt -l .)"
go test ./...
go run ./cmd/tunwarden version
go run ./cmd/tunwarden doctor
go run ./cmd/tunwarden recover
```

Expected result:

- formatting is clean;
- tests pass;
- `version` exits `0`;
- read-only `doctor` exits with healthy output or actionable warnings for the current host;
- read-only `recover` does not remove files, kill processes, or mutate networking.

## Safety snapshots

Capture before, active, rollback-failure, and final snapshots. Store the raw files outside the repository and publish only redacted summaries.

Required commands:

```bash
ip -d link show
ip route show table all
ip rule show
resolvectl status --no-pager || true
sudo nft -a list ruleset
pgrep -a xray || true
pgrep -a tun2socks || true
sudo find /run/tunwarden -maxdepth 4 -xdev -print -ls 2>/dev/null || true
```

Also capture firewall frontend state when present:

```bash
if command -v ufw >/dev/null 2>&1; then
  sudo ufw status verbose
  sudo ufw show raw
fi

if command -v iptables-save >/dev/null 2>&1; then
  sudo iptables-save
fi

if command -v ip6tables-save >/dev/null 2>&1; then
  sudo ip6tables-save
fi
```

`resolvectl status || true` is acceptable only when the host does not use `systemd-resolved`; the verification record must explicitly say whether DNS state was inspectable. Firewall and nftables snapshots must either succeed or the host is not suitable for this release-gate conclusion.

## Manual safe TUN flow

Build the current binaries so the CLI and daemon use the same commit:

```bash
mkdir -p ./bin
go build -o ./bin/tunwarden ./cmd/tunwarden
go build -o ./bin/tunwardend ./cmd/tunwardend
export TW="$PWD/bin/tunwarden"
```

Start the development privileged daemon deployment for `./bin/tunwardend`. The exact launcher is host-specific for this preview, but it must use the commit under test and must expose the daemon socket expected by the CLI.

Prepare a profile:

```bash
"$TW" import "$TUNWARDEN_ACCEPTANCE_IMPORT_TARGET"
"$TW" profile list
```

Record the selected `<profile-id>`, then run:

```bash
"$TW" doctor
"$TW" plan --mode tun <profile-id>
"$TW" plan --mode tun <profile-id> --json
"$TW" connect --mode tun <profile-id>
"$TW" status
"$TW" doctor
```

Expected active-state result:

- `connect --mode tun` succeeds without running the daemon or Xray as root;
- `status` reports an active `tun` connection;
- `status` includes a transaction summary with committed state and no cleanup requirement;
- `doctor` reports daemon, core, TUN, route, DNS, firewall, and transaction checks;
- `ip link show tunwarden0` shows the TunWarden TUN interface while connected;
- `ip route show table 51820` or the named TunWarden table shows the full-tunnel route owned by TunWarden;
- `ip rule show` shows only the planned TunWarden policy rules as new rules;
- `resolvectl status tunwarden0 --no-pager` shows the planned DNS servers and route-only domain `~.` when systemd-resolved is used;
- `sudo nft list table inet tunwarden` shows only TunWarden-owned nftables state;
- Xray and `tun2socks` are running as daemon-supervised child processes;
- generated runtime config exists only under the documented runtime layout and is not printed in logs or CLI output.

Verify that the VPN server route bypasses the TUN:

```bash
# Prefer the concrete server IP from `plan --mode tun --json` when available.
ip route get <server-ip>
```

Expected result: the route to the active server uses the physical/default uplink, not `tunwarden0`.

Then disconnect:

```bash
"$TW" disconnect
"$TW" status
"$TW" doctor
"$TW" recover
```

Expected final-state result:

- `disconnect` succeeds and is safe to repeat;
- `status` is inactive;
- `doctor` does not report stale TunWarden-owned TUN, route, DNS, firewall, core, or transaction state;
- `recover` remains dry-run and reports no cleanup requirement after a clean disconnect;
- `tunwarden0` is gone;
- TunWarden-owned routes and policy rules are gone;
- `resolvectl status tunwarden0` no longer reports active TunWarden DNS state;
- `nft table inet tunwarden` is gone unless a future documented persistent policy explicitly says otherwise;
- generated runtime config, TUN adapter process, Xray process, and transaction files from the connection are gone.

## Rollback failure check

Run at least one post-apply failure check on a disposable Tier 1 host or VM. The safest required scenario is an Xray startup failure after the daemon has applied the network transaction.

Use an executable Xray failure shim that exits with a non-zero status immediately, and point the development daemon at it with `TUNWARDEN_XRAY_PATH`. Keep `TUNWARDEN_TUN2SOCKS_PATH` pointed at the real adapter binary. Then run:

```bash
"$TW" connect --mode tun <profile-id>
"$TW" status
"$TW" doctor
"$TW" recover
```

Expected result:

- `connect --mode tun` fails clearly;
- the error states that applied TunWarden-owned networking state was rolled back, or it provides an actionable failure explaining what still requires cleanup;
- final snapshots show no lingering `tunwarden0`, TunWarden policy rules, TunWarden routes, per-link DNS on `tunwarden0`, `table inet tunwarden`, generated runtime configs, Xray shim process, TUN adapter process, or transaction files requiring cleanup;
- plain `recover` remains non-mutating and only reports current recovery candidates;
- if rollback is incomplete, the release gate fails until `recover --execute --yes` or a code fix restores the documented cleanup contract.

After this failure check, restart the daemon with the real Xray binary before repeating the success flow.

## Explicit recovery dependency

Before closing the v0.2 release gate, verify that the dedicated explicit recovery issue has already passed its own acceptance checks.

At minimum, the release record must link the recovery PR or issue and confirm:

```bash
"$TW" recover
"$TW" recover --execute --yes
"$TW" recover --execute --yes --json
```

Expected result:

- `recover` without `--execute` is always read-only;
- `recover --execute --yes` is daemon-owned cleanup, not CLI-side privileged mutation;
- ambiguous resources are skipped and left unchanged;
- stale PID metadata alone is not used to signal processes;
- transaction state is removed only after cleanup succeeds;
- JSON reports `status: "fail"` for failed cleanup and `status: "warn"` for incomplete cleanup.

## Known v0.2 preview limitations

These limitations are acceptable only when visible in release notes:

- v0.2 is a safe TUN preview, not a stable laptop VPN release;
- suspend/resume, Wi-Fi roaming, reconnect loops, and NetworkManager event handling are deferred;
- packaged privileged daemon deployment is not accepted by this checklist;
- non-systemd DNS fallback and configurable DNS policy are deferred;
- IPv6 full-tunnel and leak handling remain explicit future work unless separately verified;
- this checklist requires one Tier 1 host and does not replace later network-namespace, VM, and long-run reliability testing.
